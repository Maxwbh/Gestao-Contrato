Fase 5 — Notificações & Templates

Sistema de envio de mensagens multicanal

Concluído
Template Unificado Email + SMS + WhatsApp
Concluído

Um único registro de template por tipo de evento (Boleto Criado, Vence amanhã, etc.), com campos separados para cada canal: corpo_html (e-mail HTML), corpo (SMS, máx. 255 chars) e corpo_whatsapp (WA markdown). Canal determinado automaticamente pelos campos preenchidos — sem campo "Canal de Envio". Migration de dados mescla registros separados existentes automaticamente.

Migration 0005 3 canais / 1 registro Concluído
Editor HTML Avançado (TinyMCE 5)
Concluído

Formulário de templates reescrito com TinyMCE 5 (self-hosted, sem API key) para edição WYSIWYG do corpo HTML do e-mail. Contador de caracteres SMS em tempo real com substituição de %%TAGs%% por valores de exemplo para estimar o comprimento real da mensagem antes do envio.

TinyMCE 5 SMS ≤ 255 Tag substitution counter Concluído
TEST_MODE Safeguard + Phone Normalization
Concluído

Em TEST_MODE=True, todos os e-mails e SMS são redirecionados para endereços de teste configuráveis — nunca chegam ao comprador real. Celular no formato (31) 99999-8888 normalizado automaticamente para E.164 (+5531999998888) exigido pelo Twilio.

TEST_MODE E.164 normalization Twilio SMS Concluído
PDF Boleto Persistido no Banco de Dados
Concluído

Campo boleto_pdf_db (BinaryField) armazena o PDF do boleto no PostgreSQL, garantindo disponibilidade após redeploy no Render (que usa armazenamento efêmero). Fallback em 3 camadas: disco → banco → regeneração.

BinaryField PostgreSQL Render ephemeral fix Concluído
