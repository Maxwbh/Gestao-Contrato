# HU-05 — Reajuste de Parcelas

## História do Usuário

**Como** operador da imobiliária  
**Quero** aplicar o reajuste econômico anual às parcelas do contrato  
**Para** manter o equilíbrio financeiro conforme o índice (IPCA, IGPM etc.) acordado em contrato

---

## Pré-condições

- Contrato com `tipo_correcao ≠ FIXO`
- `IndiceReajuste` cadastrado para o tipo e período do ciclo alvo
- Ciclo anterior já aplicado (sequência obrigatória — não pula ciclos)
- Usuário com permissão de aplicar reajuste

---

## Critérios de Aceitação

1. `GET /financeiro/contrato/<id>/reajuste/preview/` → JSON com tabela simulada por parcela (valor_atual → valor_novo) **sem persistir**
2. `POST /financeiro/contrato/<id>/reajuste/api/` → aplica o reajuste em transação atômica
3. Período de referência calculado automaticamente:
   - `início = data_contrato + (ciclo − 2) × prazo + 1 mês`
   - `fim = data_contrato + (ciclo − 1) × prazo`
4. Cálculo do percentual:
   - Método 1 (prioritário): `% = numero_indice_fim / numero_indice_base − 1` (número-índice IBGE/FGV)
   - Método 2 (fallback): produto das variações mensais `∏(1 + valor_k/100) − 1`
   - Aplica spread: `percentual_bruto_com_spread = percentual_bruto + spread_reajuste`
   - Aplica desconto: `percentual_liquido = max(bruto_spread − desconto_percentual, piso_reajuste)`
   - Aplica teto: `percentual_final = min(percentual_liquido, teto_reajuste)`
5. **Modo simples** (sem `TabelaJurosContrato`): `novo_valor = valor_atual × (1 + percentual_final/100)`
6. **Modo Tabela Price** (com `TabelaJurosContrato`):
   - `saldo = SUM(valor_atual)` de parcelas NORMAL não pagas
   - `saldo_atualizado = saldo × (1 + percentual_bruto/100)`
   - `novo_pmt = PMT(saldo_atualizado, juros_ciclo_n, n_restantes)`
   - Novo PMT aplicado a todas as parcelas restantes
7. Fallback de índice: se índice principal falha, tenta `tipo_correcao_fallback`
8. Após aplicação: `contrato.ciclo_reajuste_atual = ciclo`; boletos já emitidos do ciclo são regenerados com novo `valor_boleto`
9. Registro `Reajuste` criado com campos de auditoria: `usuario`, `ip_address`, `data_aplicacao`
10. `GET /financeiro/reajustes/pendentes/` exibe **Grid de Reajustes Pendentes** com as colunas:
    - Contrato / Data · Comprador / Imóvel · Ciclo / Parcelas
    - Índice / Período de Referência (via `calcular_periodo_referencia()`)
    - Prestação Atual · Correção % (via `IndiceReajuste.get_acumulado_periodo()`) · Prestação Nova calculada
    - Ações: Aprovar (individual) + Editar (individual com modal %)
11. `POST /financeiro/reajustes/aplicar-lote/` → aplica índice calculado a múltiplos contratos (**Aplicar Calculado em lote**)
12. `POST /financeiro/reajustes/aplicar-informado-lote/` → aplica percentual informado pelo operador a múltiplos contratos (**Aplicar Informado em lote**); campo `observacoes` opcional
13. Detecção antecipada: `calcular_ciclo_pendente(antecipacao_meses=1)` exibe contrato na grid **1 mês antes** do aniversário, independente do dia do mês

---

## Regras de Negócio

- RN-01: Ciclo 1 não sofre reajuste (parcelas geradas com valor original)
- RN-02: Ciclo N usa o índice acumulado do **período anterior** (ciclo N−1)
- RN-03: Obrigatoriedade de sequência: ciclo 3 exige ciclo 2 aplicado; violação retorna erro
- RN-04: `FIXO` nunca tem reajuste pendente e nunca bloqueia boletos
- RN-05: Reajuste pendente é detectado 1 mês antes do aniversário (via `antecipacao_meses=1`) — detecção por mês, não por dia: contrato 15/04/2024 aparece na grid em 01/03/2025
- RN-06: Campos de snapshot no registro: `spread_aplicado`, `piso_aplicado`, `teto_aplicado` preservam os parâmetros usados
- RN-07: Desconto pode ser percentual (`desconto_percentual`) ou fixo por parcela (`desconto_valor`)

---

## Definição de Pronto

- [ ] `Reajuste` salvo com `aplicado = True` e `data_aplicacao` preenchida
- [ ] Todas as parcelas do ciclo alvo com `valor_atual` atualizado
- [ ] `contrato.ciclo_reajuste_atual` incrementado
- [ ] Boletos já emitidos regenerados com novo `valor_boleto`
- [ ] Sequência de ciclos validada (não pula)
- [ ] Preview retorna tabela correta sem persistir
- [ ] Reajuste em lote (calculado) processa múltiplos contratos via `aplicar-lote`
- [ ] Reajuste informado em lote via `aplicar-informado-lote` com % único e observações
- [ ] Grid de reajustes pendentes exibe Correção %, Prestação Nova e ações Aprovar/Editar
- [ ] Detecção antecipada 1 mês antes do aniversário (`antecipacao_meses=1`)

## Cenários de Teste

### CT-01 — Preview IPCA ciclo 2 (modo simples)
**Dado** contrato com IPCA, prazo 12, data jul/2020, ciclo atual = 1  
**E** IPCA acumulado ago/2020–jul/2021 = 8,99%  
**Quando** `GET .../reajuste/preview/` para ciclo 2  
**Então** JSON com `percentual_final = 8,99%` e `valor_novo` calculado para cada parcela

### CT-02 — Aplicação Modo Tabela Price
**Dado** `TabelaJurosContrato` com ciclo 2 → 0,6%/mês, 60 parcelas restantes, saldo devedor R$ 200.000  
**Quando** aplica reajuste ciclo 2 com IPCA 8%  
**Então** `saldo_atualizado = 200.000 × 1,08 = 216.000`; `novo_pmt = PMT(216.000, 0,006, 60)` aplicado às 60 parcelas

### CT-03 — Tentativa de pular ciclo
**Dado** contrato com ciclo atual = 2, reajuste do ciclo 2 não aplicado  
**Quando** tenta aplicar ciclo 3 diretamente  
**Então** retorna erro "Ciclo 2 deve ser aplicado antes do ciclo 3"

### CT-04 — Índice principal indisponível, fallback usado
**Dado** `tipo_correcao = IGPM`, IGPM não cadastrado para o período, `tipo_correcao_fallback = IPCA`  
**Quando** calcula o reajuste  
**Então** usa IPCA como fallback e registra no campo `indice_tipo` do `Reajuste`

### CT-05 — Piso e teto de reajuste
**Dado** índice = 15%, teto = 12%  
**Quando** calcula percentual final  
**Então** `percentual_final = 12%` (teto aplicado)

## Fluxo do Processo

```
Operador acessa reajuste do contrato
    │
    ▼
GET /financeiro/contrato/<id>/reajuste/
    │   Sistema detecta ciclo pendente via:
    │   Reajuste.calcular_ciclo_pendente(contrato, antecipacao_meses=1)
    │
    ▼
GET /financeiro/contrato/<id>/reajuste/preview/  (JSON)
    │
    ├── Calcular período de referência (início/fim)
    ├── IndiceReajuste.get_acumulado_periodo() → percentual_bruto
    │       Método 1: numero_indice_fim / numero_indice_base − 1
    │       Método 2 (fallback): ∏(1 + valor_k/100) − 1
    ├── Aplicar: spread → desconto → piso → teto
    ├── MODO SIMPLES: novo = valor × (1 + pct/100)
    │   MODO PRICE: saldo_atualizado → PMT(saldo, taxa_ciclo, n_restantes)
    └── Retorna tabela: [{parcela, valor_atual, valor_novo, diferença}]
    │
    ▼
Operador confirma aplicação
    │
POST /financeiro/contrato/<id>/reajuste/api/
    │
    ├── Validar sequência (ciclo_esperado = calcular_ciclo_pendente())
    ├── Criar Reajuste(aplicado=False)
    ├── Para cada parcela do ciclo: parcela.valor_atual = novo_valor
    ├── Reajuste.aplicado = True; data_aplicacao = now()
    ├── contrato.ciclo_reajuste_atual = ciclo
    ├── Se boletos emitidos: regenerar com novo valor_boleto
    └── Retorna {sucesso, ciclo, parcelas_atualizadas, boletos_regenerados}
    │
    ▼
Reajuste aplicado ✅
```
