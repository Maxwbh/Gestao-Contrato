# HU-16 — CNAB — Remessa e Retorno Bancário

## História do Usuário

**Como** operador da imobiliária  
**Quero** gerar arquivos de remessa CNAB para registrar boletos no banco e processar arquivos de retorno com confirmações de pagamento  
**Para** automatizar a conciliação bancária em lote e garantir que os registros de pagamento estejam sincronizados com o extrato bancário

---

## Pré-condições

- Boletos com `status_boleto = GERADO` disponíveis para remessa
- Conta bancária configurada com suporte a CNAB (convênio, agência, conta)
- Arquivo `.rem` (remessa) ou `.ret` (retorno) no formato CNAB 240 ou CNAB 400

---

## Critérios de Aceitação

### Remessa

1. `GET /financeiro/cnab/remessa/` → lista todos os arquivos de remessa gerados, com status e data
2. `POST /financeiro/cnab/remessa/gerar/` → gera arquivo de remessa CNAB para boletos selecionados
   - Inclui apenas boletos com `status_boleto ∈ {GERADO, REGISTRADO}` e `nosso_numero` preenchido
   - Retorna arquivo para download e cria registro `ArquivoRemessa`
3. `GET /financeiro/cnab/remessa/<pk>/download/` → baixa o arquivo `.rem` gerado
4. `POST /financeiro/cnab/remessa/<pk>/marcar-enviada/` → marca como enviado ao banco (status `ENVIADA`)
5. `POST /financeiro/cnab/remessa/<pk>/regenerar/` → regenera o arquivo (ex: após corrigir dados)
6. `POST /financeiro/cnab/remessa/<pk>/excluir/` → exclui remessa não enviada
7. `GET /financeiro/api/cnab/boletos-disponiveis/` → JSON com boletos elegíveis para próxima remessa
8. `GET /financeiro/api/cnab/boletos-pendentes/count/` → contagem de boletos aguardando remessa

### Retorno

9. `GET /financeiro/cnab/retorno/` → lista todos os arquivos de retorno importados
10. `POST /financeiro/cnab/retorno/upload/` → faz upload de arquivo `.ret` e processa registros
11. `GET /financeiro/cnab/retorno/<pk>/` → detalhe do retorno com itens processados
12. `POST /financeiro/cnab/retorno/<pk>/processar/` → processa o arquivo de retorno:
    - Para cada registro de pagamento: concilia com parcela por `nosso_numero`
    - Cria `HistoricoPagamento` com `origem_pagamento = CNAB`
    - Atualiza `status_boleto = PAGO`
13. `GET /financeiro/cnab/retorno/<pk>/download/` → baixa o arquivo `.ret` original

---

## Regras de Negócio

- RN-01: Apenas boletos com `nosso_numero` preenchido entram na remessa
- RN-02: Arquivo de retorno é idempotente — processar o mesmo arquivo duas vezes não duplica pagamentos (valida por `nosso_numero + data_pagamento`)
- RN-03: Retorno com código de movimento "liquidado" (código 6 no CNAB 400) → pagamento confirmado
- RN-04: Retorno com código de movimento "cancelado/baixado" → `status_boleto = CANCELADO`
- RN-05: Cada banco tem seu próprio layout de campos (BB 001, Bradesco 237, Itaú 341 etc.) — processado pelo `CNABService`
- RN-06: Arquivo de retorno armazenado como binário para auditoria futura
- RN-07: `ArquivoRemessa` e `ItemRetorno` são vinculados às parcelas para rastreabilidade completa

---

## Definição de Pronto

- [ ] Remessa gerada com todos os campos obrigatórios por banco
- [ ] Download do arquivo `.rem` funcional
- [ ] Upload de retorno processa e concilia pagamentos
- [ ] Idempotência: segundo processamento do mesmo retorno não duplica
- [ ] `HistoricoPagamento` com `origem_pagamento = CNAB` criado
- [ ] Status do boleto atualizado após processamento

## Cenários de Teste

### CT-01 — Gerar remessa com 5 boletos
**Dado** 5 boletos com `status_boleto = GERADO` e `nosso_numero` preenchido  
**Quando** `POST /financeiro/cnab/remessa/gerar/`  
**Então** arquivo `.rem` gerado, `ArquivoRemessa` criado com 5 itens, disponível para download

### CT-02 — Boleto sem nosso_numero excluído da remessa
**Dado** 6 boletos, sendo 1 sem `nosso_numero`  
**Quando** gera remessa  
**Então** arquivo com 5 registros (o sem `nosso_numero` é ignorado)

### CT-03 — Processar retorno com pagamentos
**Dado** arquivo `.ret` com 3 registros de liquidação (código 6)  
**Quando** `POST /financeiro/cnab/retorno/<pk>/processar/`  
**Então** 3 `HistoricoPagamento` criados com `origem=CNAB`, 3 parcelas com `pago=True`

### CT-04 — Processar retorno duas vezes — idempotência
**Dado** retorno já processado  
**Quando** processa novamente  
**Então** nenhum `HistoricoPagamento` duplicado; aviso de registros já processados

### CT-05 — Retorno com cancelamento
**Dado** registro de retorno com código de movimento "baixado"  
**Quando** processado  
**Então** `parcela.status_boleto = CANCELADO`, sem criar `HistoricoPagamento`

## Fluxo do Processo

```
REMESSA
    │
    ▼
GET /financeiro/api/cnab/boletos-disponiveis/
    │   JSON: boletos elegíveis {parcela_id, nosso_numero, valor, vencimento}
    │
    ▼
POST /financeiro/cnab/remessa/gerar/
    │
    ├── Filtrar: status_boleto ∈ {GERADO, REGISTRADO} e nosso_numero NOT NULL
    ├── CNABService.gerar_remessa(boletos, conta_bancaria)
    │       → Monta registros no layout do banco (CNAB 240 ou 400)
    │       → Calcula dígitos verificadores
    ├── ArquivoRemessa.create(arquivo=bytes, status=GERADO)
    └── Retorna: {arquivo_id, total_registros, download_url}
    │
    ▼
POST /financeiro/cnab/remessa/<pk>/marcar-enviada/
    │   status = ENVIADA
    │
    ▼ (banco processa, retorna arquivo .ret)

RETORNO
    │
    ▼
POST /financeiro/cnab/retorno/upload/
    │   Upload arquivo .ret
    ├── ArquivoRetorno.create(arquivo=bytes, status=AGUARDANDO)
    │
    ▼
POST /financeiro/cnab/retorno/<pk>/processar/
    │
    ├── CNABService.processar_retorno(arquivo_retorno)
    │       → Parse dos registros do arquivo
    │       → Para cada registro:
    │               parcela = Parcela.objects.get(nosso_numero=nosso_numero)
    │               Se código=LIQUIDADO:
    │                   HistoricoPagamento.create(origem=CNAB, fitid=nosso_numero+data)
    │                   parcela.pago = True; status_boleto = PAGO
    │               Se código=CANCELADO:
    │                   parcela.status_boleto = CANCELADO
    │
    └── ArquivoRetorno.status = PROCESSADO
    │
    ▼
Conciliação bancária concluída ✅
```
