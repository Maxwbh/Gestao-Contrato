# HU-03 — Gerar Boleto Individual

## História do Usuário

**Como** operador da imobiliária  
**Quero** gerar um boleto bancário para uma parcela específica  
**Para** que o comprador possa efetuar o pagamento através do banco com código de barras, linha digitável e PIX

---

## Pré-condições

- Contrato `ATIVO` com parcelas geradas (HU-01 + HU-02)
- Conta bancária configurada na imobiliária (`core.ContaBancaria`)
- Parcela com `pago = False`
- API `boleto_cnab_api` disponível (Docker, porta 9292)
- Sem bloqueio de reajuste ativo para a parcela (ver HU-06), salvo `force=True`

---

## Critérios de Aceitação

1. `POST /financeiro/parcelas/<id>/boleto/gerar/` retorna JSON `{sucesso, nosso_numero, ...}`
2. Sistema verifica via `Parcela.pode_gerar_boleto()` se há bloqueio por reajuste pendente; se houver, retorna erro descritivo
3. Se boleto já existe e `force=False`, retorna o boleto existente sem gerar novo
4. Com `force=True` no payload, regenera o boleto mesmo com bloqueio ou boleto existente
5. Campos preenchidos na parcela após geração:
   - `nosso_numero`, `nosso_numero_formatado`, `nosso_numero_dv`
   - `codigo_barras`, `linha_digitavel`
   - `numero_documento`
   - `boleto_pdf` (FileField) e `boleto_pdf_db` (BinaryField — persistência em ambiente efêmero)
   - `pix_copia_cola`, `pix_qrcode` (base64)
   - `status_boleto = GERADO`
   - `data_geracao_boleto`
6. Notificação ao comprador é agendada assincronamente (não bloqueia o request)
7. `GET /financeiro/parcelas/<id>/boleto/status/` retorna JSON com status atual do boleto
8. `POST /financeiro/parcelas/<id>/boleto/cancelar/` cancela o boleto com motivo registrado
9. `GET /financeiro/parcelas/<id>/boleto/download/` serve o PDF diretamente
10. `GET /financeiro/parcelas/<id>/boleto/visualizar/` renderiza página HTML com todos os dados

---

## Regras de Negócio

- RN-01: Parcela com `pago = True` não pode ter boleto gerado
- RN-02: Se `tipo_correcao = FIXO` no contrato → `pode_gerar_boleto()` sempre retorna `True`
- RN-03: Se ciclo da parcela = 1 → sempre permitido
- RN-04: Para ciclo ≥ 2: verifica cascata — todos os ciclos de 2 até o ciclo da parcela devem ter reajuste aplicado; se qualquer um estiver pendente e vencido → bloqueia
- RN-05: Mapeamento de campos por banco:

| Banco | Código | Campo Específico |
|-------|--------|-----------------|
| Banco do Brasil | 001 | convenio (4–8 dígitos) |
| Santander | 033 | convenio (7 dígitos) |
| Bradesco | 237 | nosso_numero_len=11 |
| Itaú | 341 | carteira=175 |
| Caixa | 104 | emissao='4' |
| Sicredi | 748 | posto (2 dígitos) |
| Sicoob | 756 | variacao='01' |

- RN-06: `boleto_pdf_db` (BinaryField) garante acesso ao PDF mesmo em ambientes com storage efêmero (Render.com)
- RN-07: Cancelamento registra `motivo` e altera `status_boleto = CANCELADO`
- RN-08: **Paridade por banco (BRCobrança)** — a geração só é possível para bancos da tabela única
  `financeiro/services/bancos.BANCOS_SUPORTADOS` (mesma fonte consumida pela HU-23 para remessa).
  O código do banco vem de `ContaBancaria.banco`; o identificador no BRCobrança vem de
  `bancos.brcobranca_id(codigo)`. Banco fora dessa tabela não gera boleto.
- RN-09: **Boleto sem remessa CNAB** — alguns bancos suportados **geram boleto mas não têm layout
  de remessa** (`layouts_cnab` vazio: ex. Inter 077, Nubank 260, Safra 422, Banestes 021). Para esses,
  o registro é online (não há CNAB) e eles **não entram na remessa** (ver HU-23 RN-16). A emissão do
  boleto funciona normalmente; o que não se aplica é o registro via arquivo.

---

## Paridade por banco (BRCobrança) — HU-03 × HU-23

Três tabelas precisam permanecer coerentes; a fonte de verdade do suporte é
`financeiro/services/bancos.BANCOS_SUPORTADOS`:

| Tabela | Onde | Papel |
|--------|------|-------|
| `core.models.BancoBrasil` | seletor da `ContaBancaria` | bancos que o usuário pode escolher |
| `BANCOS_SUPORTADOS` | `financeiro/services/bancos.py` | bancos que o BRCobrança **gera boleto** (HU-03) |
| `layouts_cnab` (por banco) | mesma tabela | bancos que o BRCobrança **monta remessa** (HU-23) |

- **Boleto (HU-03):** exige banco ∈ `BANCOS_SUPORTADOS`. `BoletoService.BANCOS_BRCOBRANCA` é **derivado**
  dessa tabela (sem segunda lista divergente).
- **Remessa (HU-23):** exige `bancos.suporta_cnab(codigo, layout)` — banco com `layouts_cnab` vazio não
  gera arquivo (HU-23 RN-16) e é excluído da conferência com aviso "registro online".
- **Atenção (gap conhecido):** o seletor `BancoBrasil` é hoje **mais amplo** que `BANCOS_SUPORTADOS`
  (inclui ex.: BTG 208, Sofisa 637, Mercado Pago 323, Asaas 461, "Outros" 000). Escolher um desses
  para emissão falharia no BRCobrança. A divergência é **travada por teste**
  (`tests/unit/financeiro/test_paridade_bancos.py`) para forçar decisão consciente a cada mudança;
  um guard de validação em `ContaBancaria.clean()` fica como evolução recomendada.

---

## Definição de Pronto

- [ ] Parcela com `status_boleto = GERADO` e `nosso_numero` preenchido
- [ ] `codigo_barras` e `linha_digitavel` válidos para o banco
- [ ] `boleto_pdf_db` preenchido com conteúdo PDF válido
- [ ] `pix_copia_cola` gerado
- [ ] Download PDF retorna `Content-Type: application/pdf`
- [ ] Parcela paga retorna erro ao tentar gerar
- [ ] Bloqueio de reajuste retorna mensagem descritiva

---

## Cenários de Teste

### CT-01 — Geração bem-sucedida (caminho feliz)
**Dado** parcela não paga, conta bancária Itaú configurada, sem bloqueio de reajuste  
**Quando** `POST /financeiro/parcelas/<id>/boleto/gerar/` é chamado  
**Então** resposta `{sucesso: true, nosso_numero: "..."}`, parcela com `status_boleto = GERADO`

### CT-02 — Parcela já paga
**Dado** parcela com `pago = True`  
**Quando** tenta gerar boleto  
**Então** retorna `{sucesso: false, erro: "Parcela já paga"}`

### CT-03 — Sem conta bancária
**Dado** imobiliária sem `ContaBancaria` ativa  
**Quando** tenta gerar boleto  
**Então** retorna `{sucesso: false, erro: "Nenhuma conta bancária configurada"}`

### CT-04 — Bloqueio por reajuste ciclo 2
**Dado** parcela no ciclo 2, reajuste do ciclo 2 não aplicado, mês de vencimento já passou  
**Quando** tenta gerar boleto sem `force=True`  
**Então** retorna `{sucesso: false, erro: "Reajuste do ciclo 2 pendente"}`

### CT-05 — Force ignora bloqueio
**Dado** cenário do CT-04 mas com `force=True` no payload  
**Quando** tenta gerar boleto  
**Então** boleto é gerado normalmente

### CT-06 — Download PDF
**Dado** parcela com `boleto_pdf_db` preenchido  
**Quando** `GET /financeiro/parcelas/<id>/boleto/download/`  
**Então** resposta 200 com `Content-Type: application/pdf`

### CT-07 — Paridade de banco travada por teste (HU-03 × HU-23)
**Dado** a tabela `BANCOS_SUPORTADOS` e o seletor `BancoBrasil`  
**Quando** roda `tests/unit/financeiro/test_paridade_bancos.py`  
**Então** confere: layouts CNAB válidos, `BANCOS_BRCOBRANCA` derivado da fonte única, conjunto de bancos "boleto sem remessa CNAB" (RN-09 / HU-23 RN-16) e os bancos selecionáveis sem suporte BRCobrança — qualquer drift quebra o teste

## Fluxo do Processo

```
Operador clica "Gerar Boleto"
    │
    ▼
POST /financeiro/parcelas/<id>/boleto/gerar/
    │
    ├── Verificar parcela.pago → se True: retorna erro
    │
    ├── Verificar Parcela.pode_gerar_boleto()
    │       ├── FIXO ou ciclo 1 → sempre True
    │       └── ciclo ≥ 2 → verificar cascata:
    │               Para cada ciclo de 2 até ciclo_parcela:
    │                   Se vencido e sem Reajuste(aplicado=True) → False
    │
    ├── Se bloqueado e force=False → retorna erro descritivo
    │
    ├── Se boleto já existe e force=False → retorna boleto existente
    │
    ▼
BoletoService.gerar_boleto(parcela, conta_bancaria)
    │
    ├── _montar_dados_boleto() → payload com dados do comprador, parcela, banco
    ├── _validar_dados_boleto() → verifica campos obrigatórios por banco
    ├── POST http://localhost:9292/api/<banco>/boleto → PDF + dados
    └── _processar_resposta_sucesso():
            parcela.nosso_numero = ...
            parcela.codigo_barras = ...
            parcela.linha_digitavel = ...
            parcela.boleto_pdf_db = <conteúdo PDF>
            parcela.pix_copia_cola = ...
            parcela.status_boleto = GERADO
            parcela.save()
    │
    ├── Agendar notificação email/WhatsApp (assíncrono)
    │
    ▼
Retorna JSON {sucesso: true, ...dados do boleto...}
```
