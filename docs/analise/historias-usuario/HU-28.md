# HU-28 — Cadastro e Gestão de Usuários do Sistema (Comprador × Usuário Interno)

## História do Usuário

**Como** gestor de uma contabilidade ou imobiliária
**Quero** cadastrar os usuários da minha equipe com os acessos certos — e que
compradores entrem **apenas** pelo portal, vendo só os próprios contratos
**Para** que cada pessoa veja exatamente o que pode ver, sem contas órfãs nem
acessos concedidos fora do meu escopo.

> **Origem:** revisão do cadastro de usuários pensando na implementação
> (2026-07). Todas as sub-HUs são 🆕 **NOVAS**.

---

## Os dois tipos de usuário (conceito central)

| | 👤 **Comprador (portal)** | 🧑‍💼 **Usuário do Sistema (interno)** |
|---|---|---|
| Quem é | Cliente que comprou um imóvel | Funcionário de Contabilidade/Imobiliária |
| Entra por | `/portal/` (auto-cadastro por CPF/CNPJ) | Cadastrado por um gestor (sem auto-registro) |
| Vê | **Somente os próprios contratos** | Dados das imobiliárias a que tem acesso |
| Vínculo | `AcessoComprador` → `Comprador` → contratos; **CPF/CNPJ precisa casar com um Comprador existente vinculado a contrato** | `AcessoUsuario` (N registros): múltiplas contabilidades e 1+ imobiliárias |
| Verificação | E-mail de confirmação (`PORTAL_EMAIL_VERIFICACAO`) | Convite por e-mail (recomendado) ou senha inicial com troca obrigatória |

Os dois mundos **nunca se cruzam**: usuário com `AcessoComprador` é redirecionado
ao portal; usuário interno não enxerga telas do portal.

---

## Diagnóstico do estado atual (o que o código já tem — e os gaps)

**Já existe e será reaproveitado:**
- `AcessoUsuario` (core/models.py) — modelo N:N usuário×contabilidade×imobiliária
  com `pode_editar`/`pode_excluir`/`ativo`: **atende** "1 usuário em várias
  contabilidades e 1+ imobiliárias" sem migração nova.
- Telas de acessos (`core:listar_acessos`, criar/editar/excluir) e a API
  `api_listar_acessos_usuario`.
- Isolamento por tenant via `get_imobiliarias_usuario()` em todas as consultas.
- Portal do comprador: `auto_cadastro` localiza o `Comprador` pelo documento e
  cria `AcessoComprador`; verificação de e-mail opcional por parâmetro.

**Gaps encontrados (o que esta HU corrige):**
1. 🔴 **Auto-registro aberto de usuário interno** — `/accounts/registro/` permite
   qualquer pessoa criar conta e cair no dashboard (sem aprovação). O isolamento
   de tenant impede ver dados, mas a conta órfã existe e acessa telas internas.
2. 🔴 **Concessão de acesso sem escopo do cadastrador** — `AcessoUsuarioForm`
   lista **todas** as contabilidades/imobiliárias ativas: um usuário comum pode
   conceder (a si ou a terceiros) acesso a tenants que ele próprio não tem.
3. 🟡 **Criação de usuário desconectada dos acessos** — o usuário nasce por
   auto-registro/admin e os acessos são dados depois, em outra tela; não há um
   fluxo único "cadastrar funcionário com seus acessos".

## Sub-HUs

### HU-28.1 — Cadastrar usuário interno com acessos (fluxo único) 🆕
**Como** gestor **quero** um formulário único que crie o usuário E seus acessos
**para que** o funcionário já nasça com o escopo certo.
**Aceite:**
- Tela **Cadastros → Usuários → Novo** com: **Nome** (nome + sobrenome),
  **E-mail** (vira `username`; único), **Credencial** (ver 28.2) e **Acessos**
  (ver 28.3).
- Criação atômica (`transaction.atomic`): usuário + N `AcessoUsuario`.
- E-mail duplicado → erro claro ("já existe usuário com este e-mail") sem criar nada.

### HU-28.2 — Credencial inicial: senha definida no cadastro + flag de convite 🆕
**Como** gestor **quero** cadastrar o funcionário já com a senha inicial — e,
quando fizer sentido, apenas marcar uma flag para enviar convite por e-mail
**para que** o cadastro funcione mesmo sem depender de e-mail.
**Aceite:**
- **Padrão — Senha inicial no cadastro:** o gestor define a senha no próprio
  formulário (validadores padrão do Django) e o sistema **força a troca no
  primeiro login** (flag `PerfilUsuario.deve_trocar_senha` + middleware que
  redireciona para `alterar_senha` até a troca). A senha nunca é exibida
  depois de salva nem enviada por e-mail.
- **Flag opcional "Enviar convite por e-mail":** quando marcada, os campos de
  senha somem; o usuário é criado com `set_unusable_password()` e recebe
  e-mail com **link de definição de senha** (token do fluxo de reset do
  Django, validade 72h; reenvio disponível na listagem). Não loga antes de
  definir a senha. *(Alternativa alinhada ao padrão SaaS: comprova posse do
  e-mail e evita senha transmitida — útil para funcionário remoto.)*

### HU-28.3 — Acessos multi-tenant limitados ao escopo do cadastrador 🆕
**Como** gestor **quero** conceder acessos apenas ao que EU tenho acesso
**para que** ninguém escale privilégios via cadastro.
**Aceite:**
- O bloco "Acessos" lista **somente** as contabilidades/imobiliárias retornadas
  por `get_contabilidades_usuario(cadastrador)` / `get_imobiliarias_usuario(cadastrador)`
  (superuser/staff vê todas).
- UI em **grade agrupada por contabilidade** com checkboxes de imobiliárias e,
  por linha, `pode_editar`/`pode_excluir` (defaults: editar ✔, excluir ✖).
- **Defesa em profundidade:** o POST revalida o escopo no servidor (não confia
  na UI); tentativa fora do escopo → 403 + registro em `LogAuditoria`.
- Mesmas regras nas telas de **editar acessos** existentes (corrige o gap 2).

### HU-28.4 — Só administradores cadastram usuários (auto-registro fechado) 🆕
**Como** administrador **quero** ser o único a criar usuários internos
**para que** não existam contas órfãs nem cadastro por quem não deve.
**Aceite:**
- **Cadastro/gestão de usuários é restrito a administradores** — a tela
  (`Cadastros → Usuários`) e as ações (criar/editar acessos/desativar) só são
  acessíveis a quem é **administrador** (perfil ADMIN — ver 28.7 — ou
  superuser/staff). Usuário comum não vê o item de menu; acesso direto à URL →
  403 + `LogAuditoria`.
- `/accounts/registro/` (auto-registro aberto) desativado — redirect para login
  com mensagem; feature-flag `PERMITIR_AUTO_REGISTRO=False` (default) e o link
  "Registre-se" sai da tela de login.
- O auto-cadastro do **portal do comprador** permanece (é outro fluxo).

### HU-28.7 — Perfil do usuário: Administrador × Usuário comum 🆕
**Como** administrador **quero** definir, no cadastro, se o novo usuário é
**Administrador** ou **Usuário comum**
**para que** fique claro quem pode gerenciar usuários e quem apenas opera.
**Aceite:**
- Campo **Perfil de acesso** = **switch "Administrador"** (default **desligado**):
  - **Desligado → Usuário comum:** opera os módulos das imobiliárias a que tem
    acesso; **não** vê a tela de usuários nem gerencia acessos.
  - **Ligado → Administrador:** além de operar, **pode cadastrar e gerenciar
    usuários e conceder acessos** — sempre **limitado ao próprio escopo** (RN-2).
    Um administrador **não** cria superusuários nem escala além do que ele acessa.
- Persistência: `PerfilUsuario.papel` (`ADMIN` | `COMUM`, default `COMUM`).
  O predicado "pode gerenciar usuários" = `papel == ADMIN` **ou**
  `usuario_tem_permissao_total(user)` (superuser/staff continua podendo tudo).
- **Perfil ≠ permissões por imobiliária:** `papel` diz *se administra usuários*;
  `pode_editar`/`pode_excluir` de cada `AcessoUsuario` dizem *o que faz nos
  dados de cada imobiliária*. São dimensões independentes.

### HU-28.5 — Comprador entra só pelo portal, com documento validado 🆕
**Como** comprador **quero** me cadastrar com meu CPF/CNPJ **para que** eu veja
meus contratos — e nada além.
**Aceite (formaliza o comportamento atual + reforços):**
- Documento informado precisa **casar com um `Comprador` existente vinculado a
  pelo menos um contrato**; sem correspondência → orientação para procurar a
  imobiliária (sem revelar se o documento existe).
- Verificação de e-mail ativa em produção (`PORTAL_EMAIL_VERIFICACAO=True`).
- Conta do comprador **não** recebe `AcessoUsuario` (mundos separados);
  usuário interno **não** pode ter `AcessoComprador`. Validação nos dois fluxos.

### HU-28.6 — Gerir o ciclo de vida (desativar, reenviar convite, auditar) 🆕
**Como** gestor **quero** desativar quem saiu e auditar concessões
**para que** o acesso acompanhe o quadro real da equipe.
**Aceite:** listagem de usuários do meu escopo com status (ativo/convite
pendente), ações **desativar** (`is_active=False` — soft), **reenviar convite**
e **editar acessos**; toda concessão/revogação gera `LogAuditoria`
(quem, para quem, tenant, quando).

## Especificação do formulário (melhores práticas pesquisadas)

Tela **restrita a administradores** (HU-28.4). Layout em **uma coluna, 2
seções** (sem wizard — é um form curto). O cabeçalho exibe apenas o título e o
selo *"Somente administradores cadastram usuários"* — sem cartões explicativos
dos tipos de usuário.

```
┌─ Dados do usuário ──────────────────────────────────────────┐
│ Nome*            [____________]  Sobrenome* [____________]  │
│ E-mail* (login)  [_________________________]                │
│ Perfil de acesso*  [ ◯ ] Administrador                      │
│    Desligado = usuário comum (só opera). Ligado = pode      │
│    cadastrar e gerenciar usuários, no próprio escopo.       │
│ Senha inicial*   [____________]  Confirmar* [____________]  │
│                  (troca obrigatória no 1º login)            │
│ [ ] Enviar convite por e-mail em vez de senha               │
│     (marcada: esconde os campos de senha; o funcionário     │
│      define a própria senha pelo link recebido)             │
├─ Acessos (limitado ao seu escopo) ──────────────────────────┤
│ ▼ Contabilidade Alfa                                        │
│   [✔] Imobiliária Central   editar ✔  excluir ✖            │
│   [ ] Imobiliária Litoral   editar ✔  excluir ✖            │
│ ▼ Contabilidade Beta                                        │
│   [✔] Imobiliária Sul       editar ✔  excluir ✖            │
│                                                             │
│ ⚠ pelo menos 1 acesso é obrigatório                         │
└─────────────────────────────────────────────────────────────┘
        [Cancelar]                    [Criar usuário]
```

Práticas aplicadas (fontes ao final da HU):
- **Senha inicial exige troca no 1º login** (padrão deste sistema, por decisão
  de produto: o cadastro não depende de e-mail); o **convite por link** fica
  disponível como flag — comprova posse do e-mail e evita transmissão de senha.
- **Papéis por organização**: permissões (`pode_editar`/`pode_excluir`) são
  **por linha de acesso**, não globais — o mesmo usuário pode editar na
  Imobiliária A e só ler na B (o modelo `AcessoUsuario` já é assim).
- **Administração self-service com escopo**: o gestor do tenant convida,
  ajusta papéis e desativa a própria equipe — nunca além do próprio escopo.
- E-mail como login (sem "username" separado); validação inline; mínimo de
  campos; erro de duplicidade apontado no próprio campo.

## Regras de negócio (resumo)

| # | Regra |
|---|---|
| RN-1 | **Só administradores** (perfil ADMIN ou superuser/staff) cadastram/gerenciam usuários; auto-registro aberto desativado |
| RN-2 | Concessão de acesso limitada ao escopo do cadastrador — validada no servidor |
| RN-3 | Pelo menos 1 acesso ao criar usuário interno |
| RN-4 | Comprador: documento deve casar com `Comprador` vinculado a contrato |
| RN-5 | `AcessoComprador` e `AcessoUsuario` são mutuamente exclusivos |
| RN-6 | Padrão: senha inicial no cadastro ⇒ troca obrigatória no 1º login. Flag de convite ⇒ sem login até definir a senha pelo link |
| RN-7 | Toda concessão/revogação/desativação registra `LogAuditoria` |
| RN-8 | Perfil (`papel` ADMIN/COMUM) é independente das permissões por imobiliária (`pode_editar`/`pode_excluir`) |
| RN-9 | Administrador não cria superusuário nem concede acesso fora do próprio escopo |

## Referências técnicas

| Área | Onde |
|---|---|
| Perfil ADMIN/COMUM | novo `PerfilUsuario.papel`; predicado "pode gerenciar usuários" = `papel==ADMIN or usuario_tem_permissao_total(user)` |
| Restrição da tela a admins | mixin/decorator em `core:*_acesso*` e na futura tela de usuários; item de menu condicional em `base.html` |

| Telas de acesso atuais | `core/views.py` (`AcessoUsuario*View`), `core/forms.py` (`AcessoUsuarioForm`) — ganham escopo |
| Auto-registro a fechar | `accounts/views.py::registro_view`, `accounts/forms.py::RegistroUsuarioForm` |
| Portal do comprador | `portal_comprador/views.py::auto_cadastro`, `AcessoComprador` |
| Convite/definição de senha | fluxo `PasswordResetTokenGenerator` do Django (token com validade estendida p/ convite) |
| Troca obrigatória | novo `PerfilUsuario.deve_trocar_senha` + middleware leve |
| Auditoria | `core/models.py::LogAuditoria` |
